I Taught My IDE to Think Like My Team

Before you start — model matters

This pattern relies on strong reasoning models. The orchestrator follows multi-step classification logic, constructs structured prompts, and chains outputs across agents. That only works reliably with models built for extended reasoning — not basic completion or chat models.

Best results with: Claude Opus 4.6, OpenAI Codex 5.3, o3-pro, Gemini 2.5 Pro — models that plan, self-correct, and maintain context across long tool-calling sessions.
Expect degraded results with: GPT-4o-mini, Claude Haiku, or other lightweight models. They'll handle simple lookups but struggle with multi-agent orchestration, disambiguation, and chained workflows.
Rule of thumb: if the model can't reliably follow a 2-page instruction set with branching logic, it won't drive this architecture well.

1. The Friction Nobody Talks About

If you work on a modern data or engineering team, your day probably looks something like this: you open a browser with 8 tabs — your cloud workspace in one, a CI/CD portal in another, a project tracker in a third, email, chat, calendar, maybe a documentation wiki. You spend the first 20 minutes of your morning just remembering where you left off, checking what happened overnight, and mentally stitching context across all those surfaces.

Nobody calls this "work." It's the tax you pay to do work. Context-switching, portal-hopping, GUID-hunting, copy-pasting IDs between tabs. Research shows this is one of the top destroyers of engineering velocity. And it's invisible — you don't clock it as lost time because it feels like preparation.

I was losing hours every week to this friction. Not because the tools were bad — individually, they're all excellent. But the seams between them were where my productivity went to die.

2. The Side Experiment

It started on a Saturday. I was frustrated after a week of 12-tab afternoons and thought: what if I could just tell VS Code what I want — in plain English — and it would figure out which tools to hit, what APIs to call, and how to stitch the results together?

Not a chatbot. Not a code autocompleter. A system that actually understands my workflow — the tools I use, the environments I work in, the safety rules my team follows — and can execute multi-step tasks across all of them from a single prompt.

I created a Markdown file. Wrote some instructions describing how I'd want an orchestrator to reason about my requests. Added a second file for a specialist agent that knew my cloud workspace. Connected it to an API via an MCP server. Typed "list the items in my dev workspace" into the VS Code chat panel.

It worked. And I immediately thought: okay, what else can I delegate?

That weekend scaffold became PM-Assist — a multi-agent system that now handles everything from morning triage to deployment validation to drafting status emails. It's all Markdown and YAML, checked into Git, and it gives every teammate the same operational capability from day one.

3. The Thought Process — Why Not Just Use What Exists?

Fair question. There's a growing ecosystem of AI agent tools. If you browse awesome-copilot or similar curated lists, you'll find dozens of Copilot extensions, agent frameworks, and automation plugins. Many of them are genuinely great — MCP servers for specific platforms, code generation agents, standalone chat assistants.

But here's what I noticed when I surveyed the landscape:

What exists vs. what I needed
Typical agent setup
  • Single agent, single domain
  • Great at one thing, blind to everything else
  • No cross-domain context sharing
  • Safety is "be careful" — no codified guardrails
  • Routing = keyword matching or user picks the right tool
  • Institutional knowledge stays in people's heads
What real cross-domain work needs
  • Multiple specialists, one conversation
  • An orchestrator that reasons about which specialist to call
  • Context flows between agents (email → tasks → status)
  • Safety guardrails enforced in config, not memory
  • Routing = signal extraction, scoring, disambiguation
  • Institutional knowledge in version-controlled files

The gap wasn't "no tools exist." The gap was nobody had built the thinking layer that sits above the tools — the orchestration logic that decomposes your intent, picks the right specialist, constructs a high-quality handoff, and stitches the results back together. That's what PM-Assist is.

4. Giving AI Its Own Thinking Pattern

This is the core idea, and it's what makes the whole system work. Instead of building a flat routing table ("if the user says 'deploy,' send to the deploy agent"), we gave the orchestrator a genuine reasoning pipeline:

How the orchestrator reasons through a request
1
Signal Extraction — pulls platform keywords, action verbs, environment scope, entity names from the prompt
2
Agent Scoring — scores every specialist 0–1 based on how many of its triggers matched the extracted signals
3
Disambiguation — when two agents could both claim the task, applies tiebreaker rules based on secondary context
4
Prompt Construction — builds a structured handoff: objective, context, skill hint, expected output format, and constraints
5
Delegation & Synthesis — hands off to the specialist, waits for the result, and merges outputs if multiple agents were involved

The beauty is that this entire thinking pattern is described in a Markdown file. It's not compiled code — it's a set of reasoning instructions that the LLM follows. You can read it, edit it, version it, and improve it sprint over sprint. When the model gets smarter, the thinking pattern gets more reliably executed. When you discover a new edge case, you add a disambiguation rule in plain English.

The orchestrator triages. Specialists strategize. Skills execute. The execution layer connects to the outside world. Nobody steps on anyone else's toes. Replace any layer independently — swap out a specialist, add a new tool, upgrade the model — and the rest keeps working.

6. Beyond MCP — CLI, APIs, and Programmatic Toolkits

MCP (Model Context Protocol) servers are fantastic for giving agents structured access to platforms. They offer a clean, standardized interface. But here's the thing people miss: MCP is just one execution channel. The real power comes from letting your agents use anything that can be invoked programmatically.

The full execution toolkit
M
MCP Servers — structured tool interface for platforms (your cloud workspace, project tracker, email/calendar). Great for CRUD operations with defined schemas.
C
CLI Tools — command-line interfaces that agents run in the terminal. Package managers, deployment CLIs, git, cloud CLIs. Agents can reason about output and chain commands.
A
REST APIs — direct HTTP calls via terminal scripts. When there's no MCP server for your niche platform, the agent can write and execute a request on the fly.
S
SDK & Scripts — Python/Node/PowerShell scripts the agent generates and executes in-session. Perfect for data transformations, validation logic, or custom automation with no pre-built tool.

Here's a concrete example: say your skill needs to export a solution from one environment and import it into another. There's no MCP tool for that exact flow — but there's a CLI that does it. The agent reads the skill procedure, reasons through the steps, runs the CLI commands in the terminal, parses the output, and reports back. It's not hardcoded — the agent figured out how to chain those commands because the skill described the intent and the agent had access to a terminal.

This is the unlock most people miss. You're not limited to the tools someone pre-built for you. If a human can do it in a terminal, an agent can reason about it, build it, and run it.

7. Your Personal Context Is the Superpower

Here's what makes this fundamentally different from a generic AI assistant: PM-Assist knows your world. Not the internet's world — yours.

Personal context that transforms generic AI into your AI
  • Environment catalog — your workspace IDs, environment names, which ones are read-only, which allow writes. The agent says "DEV" and instantly resolves it to the right endpoint.
  • User context — your default project path, your manager's name, your team's sprint cadence. Tasks get created in the right place without you specifying it every time.
  • Safety guardrails — your team's rules. "Never write to production without confirmation." "Always validate after promotion." These are enforced, not suggested.
  • Platform connections — MCP servers, CLIs, and API tokens for your specific tools. The agent accesses your data, not hypothetical data.
  • Institutional memory — session checkpoints, past decisions, known failure patterns. The agent remembers what happened in the same conversation and builds on it.

This is configured in simple YAML files. When a new team member joins, they clone the repo, fill in their personal context file with their own defaults, and immediately get the benefit of the entire team's operational knowledge — with their personal preferences applied. The senior engineer's hard-won understanding of "which environment is safe" and "where things go in the tracker" is now a config file, not tribal lore.

8. Real Workflows That Changed

The big wins aren't the quick lookups. They're the end-to-end workflows that used to span hours and multiple contexts. Here's what delegation looks like when the system actually thinks:

End-to-end notebook development & deployment

One prompt: "Create a notebook that processes the bronze-layer customer data, deploy it to our dev workspace, and configure the pipeline run."

DevOps Agent scaffolds the notebook with schema-aware boilerplate
DevOps Agent deploys to dev workspace via API
DevOps Agent configures the pipeline trigger and kicks it off
Orchestrator returns: notebook created, deployed, pipeline running — you handle the manual functional testing

The agent handles scaffolding, deployment, and wiring. You test the business logic — the part that actually needs your brain.

Promote, validate, and raise a PR

One prompt: "Promote the latest changes to UAT, validate schema parity, and open a PR for the release branch."

DevOps Agent triggers the deployment pipeline to UAT
DevOps Agent runs post-deploy validation: schema diff, row counts, metric checks
Work Items Agent creates a PR with the validation summary as the description
Orchestrator returns: promoted, all checks passed, PR #1234 ready for review

You review and approve the PR. The sequential grunt work — trigger, wait, validate, document — was delegated.

Cross-environment semantic model validation

One prompt: "Compare my models across DEV, UAT, and PROD — flag any schema drift or metric mismatches."

DevOps Agent fetches schemas from all three environments via API
DevOps Agent runs comparison queries against each environment
DevOps Agent produces structured diff: PASS / WARN / FAIL per entity
Orchestrator returns: "DEV ↔ UAT in sync. UAT ↔ PROD: 2 columns added in DEV not yet in PROD."

The old way: open three browser tabs, write and run queries manually in each, paste into a spreadsheet, diff by eye. ~45 min. Now ~5.

Morning triage — the parallel fan-out

One prompt: "Daily triage"

Orchestrator recognizes composite pattern → fans out to 3 agents in parallel
Comms Agent
email, calendar, chats
Work Items Agent
sprint board status
DevOps Agent
pipeline & job health
Orchestrator merges all results into one structured briefing in ~2 minutes

23 minutes of tab-hopping → a summary you read over your coffee.

9. Getting Started — A Minimal Scaffold

You don't need to replicate anyone's exact setup. Here's the skeleton that makes the pattern work for any domain:

your-assist/
  .vscode/
    mcp.json                 # MCP server connections (your platforms)
  agents/
    orchestrator.agent.md    # The thinking pattern — intent classification + routing
    specialist-a.agent.md    # Your first domain specialist
  skills/
    my-first-skill/
      SKILL.md               # A self-declaring procedure the agent activates
  config/
    catalog.yaml             # Environments, IDs, guardrails
    user-context.yaml        # Personal defaults (your team, your paths)
  setup.ps1                  # One-command onboarding

Start small. One orchestrator, one specialist, one skill. Connect an MCP server (or just a CLI) to whatever platform you use most. Write the thinking pattern in the orchestrator's Markdown — how should it reason about requests? What signals indicate this agent? Add specialists as new domains emerge. The whole thing is text files in Git. No runtime, no infra, no vendor lock-in. You iterate on it every sprint like any other code.

10. The Honest Caveats

Let me be real: it doesn't work every time.

Sometimes the model misroutes a request to the wrong agent. Sometimes an API call times out. Sometimes the agent produces a result that needs a second pass or misses a nuance that a human would catch. This is a probabilistic system, not a deterministic CI/CD pipeline. If you expect 100% reliability, you'll be frustrated.

But here's the reframe that changed my perspective:

When it doesn't work, you've lost nothing. It's a failed background session — you're exactly where you started. When it does work — which is the majority of the time — you've just saved 20–40 minutes of tedious, error-prone manual work. The expected value math is overwhelmingly positive. You're betting minutes of wait time against hours of saved effort.

The failure mode is "I'll do it the old way this time." The success mode is "I just got 30 minutes of my life back." That's a bet I'll take every single day.

11. This Isn't Just for DevOps — It's for Any Domain

Everything I've described is domain-agnostic. The pattern is: orchestrator + specialists + skills + execution layer. Swap the domain, and it still works. The thinking pattern stays the same — only the specialist knowledge changes.

Excel-DevOps

Finance teams managing spreadsheets, reconciliation workflows, compliance reports. Agent validates formulas, pushes to shared drives, flags anomalies in month-end closes.

Analytical-DevOps

BI teams managing dashboards, semantic models, data refresh schedules. Agent validates schemas, compares environments, deploys reports, and raises PRs.

ML-DevOps

Data science teams managing model training, experiment tracking, model registry. Agent compares runs, promotes champion models, updates serving endpoints.

Sales-Ops

Revenue teams managing CRM updates, pipeline reviews, forecasting. Agent pulls deal status, flags at-risk accounts, drafts executive summaries from board data.

Security-DevOps

Security teams managing access reviews, compliance scans, incident response. Agent audits permissions, flags policy violations, generates remediation tickets.

Your-Domain-DevOps

Whatever tools you use. Whatever platforms you touch. Write the specialist, connect the APIs, define the skills. The orchestration pattern doesn't change.

The idea is simple: if your job involves operating across multiple systems and contexts, you can build an assist layer that absorbs the friction. The specialist knows your domain. The orchestrator knows how to delegate. You know what actually matters.

12. Where This Is All Heading

Let's zoom out. AI isn't hype anymore. If you've been skeptical, I get it — the last few years had a lot of noise. But something shifted in a fundamental way. Models like Claude Opus 4.6 and OpenAI Codex 5.3 aren't autocomplete engines — they're genuine reasoners. They plan multi-step execution, self-correct when a tool call fails, maintain context across 50+ back-and-forth exchanges, and follow branching instruction sets that would have confused even frontier models two years ago. Opus 4.6 routinely follows a 3,000-word orchestrator definition with scoring logic, disambiguation tables, and composite workflow patterns — and gets it right. Codex 5.3 generates, executes, and iterates on code in a single session without losing the thread. This is the generation of models that makes agentic workflows practical, not theoretical. Two years ago, building PM-Assist would have been a research project. Today it's a weekend scaffold that you refine every sprint.

We're entering an era where the interface between humans and complex systems becomes a conversation. Not a dashboard. Not a portal. Not a ten-step runbook. A conversation — where you express intent, the system reasons about execution, and you review the result.

The teams that figure out how to structure that conversation well — how to give AI the right thinking patterns, the right guardrails, the right personal context — will compound their velocity in ways that feel unfair to everyone else.

The models will keep getting stronger. The execution tools will keep expanding. But the pattern — orchestrator, specialists, skills, personal context — that's the durable architecture. It doesn't depend on any one model or vendor. It adapts as the capabilities underneath it improve.

The Takeaway

AI agents aren't useful because they're clever. They're useful because they absorb the cognitive overhead that was silently eating your best hours. The delta isn't "AI vs. human" — it's human with a reasoning layer vs. human with 12 browser tabs.

Build the thinking pattern first. Define how the orchestrator should reason. Add specialists for your domains. Connect them to your tools — MCP servers, CLIs, APIs, whatever you've got. Inject your personal context. Let the agents inherit the institutional knowledge.

Start small. Iterate. And don't worry about perfection — when it fails, you lose a few seconds. When it works, you get a working day back every week.
